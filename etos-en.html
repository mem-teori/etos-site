Governance-native decision infrastructure

ETOS

Constitutionally guided self-correction in high-risk systems

ETOS is an inspectable, constitutionally guided recursive self-correction architecture. It is designed as a governance layer on top of high-risk decision systems, so tensions, explainability, historical responsibility, and learning can be handled together — without breaking governance.

Why ETOS is different

ETOS does not treat decisions as answers — but as tensions

Instead of optimising directly toward output, ETOS works with tension families, drift over time, and persistent ethical consequences. That makes it possible to say no to improvements that are technically valid if they undermine the system's normative purpose.

ETOS-Self

Proposal engine

Generates proposals from context, conflict analysis, and governance state. The output is not only a proposal, but also a conflict profile, decision trace, and governance adjustments.

CDP

Constitutional evaluation

Proposals pass through three layers: Technical, Risk, and Conceptual. This allows ETOS to reject changes that look efficient but quietly change the ruler itself.

Drift Memory

Historical awareness

The system logs direction, velocity, acceleration, oscillation, and ethical debt. ETOS therefore sees not only the present situation, but its own movement over time.

Governance Layer

Active regulation

Governance is not documentation added afterwards. It shapes behaviour through modes, intervention logic, debt lifecycle, and the Drift Pressure Index.

Recursive decision flow

A closed governance loop

ETOS is built so that future behaviour is continuously shaped by earlier decisions — but under constitutional constraints. That is the central difference between raw self-improvement and guided self-correction.

01

Input and context

The system receives a context containing tensions, conflicts, drift signals, and governance state.

02

ETOS-Self

The proposal engine generates a change proposal and a conflict profile based on context and history.

03

CDP

Technical → Risk → Conceptual review determines whether the proposal is compatible, ambiguous, or must be rejected.

04

Reflection

The decision, flags, ethical debt, and drift signals are logged in Drift Memory & Reflection.

05

Future behaviour

Feedback changes later proposals, recommended modes, and the ability to stabilise or pause the system.

Platform maturity

From drift-aware governance to active steering

The move from v3.0 to v3.5 is not just a matter of adding features. It is the story of governance becoming more operational, more historically aware, and more active in shaping the system's own development.

v3.0 → v3.1

History + feedback

Drift Pressure Index, ethical debt lifecycle, and persistent feedback memory make the system historically aware and able to learn from its own rejections.

v3.2 → v3.4

Observability + motion

Dashboards, feedback logs, and drift visualisation make the system readable. Velocity and acceleration awareness make it possible to understand not only direction, but movement.

v3.5

Active steering

Governance now recommends operational modes, can force human review, and can actively stabilise the system. Governance becomes steering, not observation alone.

AGI perspective

A practical alignment testbed on the path toward AGI

ETOS is not AGI. It is narrow and decision-focused. But it is a mature example of inspectable recursive self-correction, where the system can say no to its own improvement and show why.

Alignment gap

Changing the ruler

ETOS-Self proposes changes, but CDP defines what counts as acceptable improvement. In this way, the system tries to prevent the normative baseline from drifting along with optimisation.

Drift detection

Value drift over time

Drift Memory, oscillation detection, and unresolved debt make value drift observable. This is a practical response to a problem that most RSI projects still cannot show openly.
